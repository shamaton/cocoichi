- 分析方法: 評価分布の集計、キーワードによる論点抽出、代表レビューの目視確認

## 結論

現状の最大課題は「注文を最後まで安心して完了できないこと」です。  
その根本要因として、`注文導線の不安定さ`、`アプリ会員と注文導線の分断`、`会員登録/ログインの詰まり` が重なっており、ユーザーは「使いづらい」ではなく「アプリの存在意義が分からない」と感じています。

改善の優先順位は次の通りです。

1. 注文完了率を落とす不具合の修正
2. アプリとWebのアカウント/セッション統合
3. 会員登録の必須項目と導線の簡略化
4. クーポン利用導線の再設計
5. 継続利用したくなるポイント/特典設計

## 改善すべき点

### 1. 注文完了率を最優先KPIにして、注文中断と決済失敗を潰す

- 低評価レビューのうち `211件` が注文・決済まわりに言及
- 主な不満:
  - 決済で落ちる
  - 再注文できない
  - カート投入や注文内容が欠落する
  - 受取時間が途中で無効になりやり直しになる
  - 注文に時間がかかりすぎる

代表的な声:

> 「カード決済すると必ず落ちる。落ちた後再注文が時間がたたないと出来ない」

> 「2商品買ったのに一つだけしか購入できてない」

改善案:

- 注文作成から決済完了までの処理を idempotent にし、二重送信・中断復帰に対応する
- 決済失敗時は注文状態を明示し、`再試行` と `前回の内容でやり直す` を分ける
- カート内容と入力途中データを端末側でも保持する
- 受取時間は入力途中で剥奪せず、最終確認時に再提示する
- `注文開始数 -> 決済到達数 -> 注文確定数` のファネル監視を入れる

### 2. アプリ会員と注文導線を統合し、「結局ブラウザに飛ぶ」を解消する

- `48件` が外部ブラウザ/WEBビュー遷移を明示的に批判
- `39件` が二重会員登録やアカウント分断を示唆

代表的な声:

> 「何がなんでも結局ブラウザに飛ばすアプリ。存在する意味は何？」

改善案:

- アプリ会員とモバイルオーダー会員を統合し、認証基盤を一つにする
- 最低でも SSO とセッション共有を行い、アプリからWebへ遷移しても再ログインを発生させない
- 注文体験の主要フローはネイティブ内で完結させる
- `アプリでできること` と `Webに出す理由` を整理し、情報設計を再定義する

### 3. 会員登録を簡略化し、登録前に詰まるポイントをなくす

- 低評価レビューのうち `196件` が登録・ログイン系に言及
- `37件` がパスワード制約への不満
- `9件` が「お気に入り店舗必須」を直接批判

代表的な声:

> 「届いたURLタップしても、画面は真っ白で、それ以上進めない」

> 「アカウントの登録に近くの店舗の登録が必要なのに、GPSの読み込みが終わらない」

改善案:

- 登録時の必須項目を最小化し、お気に入り店舗登録を後回しにする
- メールリンク遷移失敗、GPS取得失敗、店舗検索失敗の各フォールバックを用意する
- パスワードルールは標準的な仕様へ寄せ、入力中に即時フィードバックを返す
- 新規登録、既存会員ログイン、注文開始の3導線を明確に分離する

### 4. クーポン導線を再設計し、「使えないクーポン」に見えないようにする

- `74件` がクーポンの使いにくさに言及
- 不満の中心:
  - モバイルオーダーで使えない
  - 店内タブレット注文との併用条件が分かりにくい
  - 使い方説明を読んでも誤解しやすい

代表的な声:

> 「クーポン目当てでアプリ導入は絶対にやめましょう。使わせる気は全くありません」

改善案:

- クーポン一覧に `利用可能な注文方法` と `利用条件` を先に表示する
- 注文フローの中で使えるクーポンを自動表示し、可能なら自動適用する
- 店内注文、テイクアウト、事前決済で仕様が違うならUIを分ける
- クーポン利用率、表示率、失敗率をイベントで追い、改善対象を定量管理する

### 5. 店舗検索と初回設定を安定させる

- `60件` が店舗選択・店舗検索系の詰まりに言及
- 郵便番号検索、GPS取得、お気に入り店舗登録で先に進めないケースがある

改善案:

- GPS失敗時は手入力検索を確実に機能させる
- 店舗検索に `現在地 / 駅名 / 郵便番号 / 店名` の複数入口を用意する
- 初回注文時にだけ一時的な店舗選択を許容し、会員設定とは分離する
- 店舗検索APIとクライアントのタイムアウト/リトライ設定を見直す

### 6. アプリの継続利用価値を強化する

- `38件` が「メリットがない」「お得感がない」と評価
- 一方で高評価では `クーポン` と `スクラッチ` が数少ない継続理由になっている
- ポイント連携、とくに楽天ポイント連携の改善要望も確認できた

代表的な声:

> 「スクラッチのイベントが始まったのでやっとアプリを開く需要ができました」

改善案:

- 常設の来店/注文インセンティブを置く
- 楽天ポイント付与や事前決済時の連携を改善する
- キャンペーン依存ではなく、通常利用でも得になる設計にする
- 注文履歴からの再注文、お気に入りメニュー保存などリピート支援を強化する

## 維持すべき点

低評価が大半ではあるものの、次の点は評価されています。

- クーポンやスクラッチ施策そのものには需要がある
- モバイルオーダー自体のニーズは高い
- 一部ユーザーは「以前より改善した」「注文方法は分かりやすい」と感じている

つまり、アプリの方向性が否定されているわけではなく、`基本体験の品質不足` が価値を打ち消している状態です。
