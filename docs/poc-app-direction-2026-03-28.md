## このPoCで検証すること

このPoCは、業務システムや注文基盤の実装を検証するものではありません。
`ネイティブアプリとして、注文体験をどこまで気持ちよく再設計できるか` を検証するためのものです。

検証テーマは次の4つです。

1. 店舗でメニューを考える感覚に近い、自然な注文体験を作れるか
2. 既存アプリで不満の大きかった操作を、ネイティブUIでどこまで解消できるか
3. 最新のUI表現やアニメーションを導入しても、使いやすさを損なわないか
4. バックエンドがなくても、体験価値の核をPoCとして成立させられるか

## 前提

### スコープ内

- モバイルアプリとしてのUX/UI設計
- 注文フローの再設計
- お気に入り保存、クーポン提案、完了演出などの体験設計
- ネイティブらしい画面遷移、アニメーション、フィードバック
- ダミーデータまたはローカルデータによるPoC実装

### スコープ外

- 実運用のバックエンドサービス構築
- 実決済連携
- 実店舗在庫連携
- 実アカウント基盤との統合
- WebView中心の実装

このPoCでは、`注文できた感覚` と `注文したくなる感覚` を作ることを優先し、実際の受注連携は扱いません。

### PoC価格方針

- アプリ内の表示価格と支払額は、店内飲食とテイクアウトで同額として扱う
- 軽減税率や税区分差は、PoC の価格体験では分岐させない
- レシート上では税区分の違いが分かる表現を許容するが、ユーザーがアプリ内で見る支払額は常に同じにする
- メニューの元データには `現在の実値` と `PoC用の仮想表示価格` を併記する
- `PoC用の仮想表示価格` は `実値に 8% を上乗せし、10円単位で切り上げた価格` を基準にする
- 目的は税計算の再現ではなく、`最後に金額が増えない` ことと `価格の分かりやすさ` を優先した体験検証にある

## PoCの目的

### 1. 既存アプリの失敗体験をUIレベルで置き換える

既存レビューでは、次の体験が強く嫌われています。

- 注文途中で止まる
- ログインや会員導線が邪魔になる
- クーポンが分かりにくい
- 注文できたかどうか不安になる
- アプリなのにWebへ飛ばされる

PoCでは、これらを次の体験へ置き換えます。

- 迷わず進める
- 途中保存される
- 最後に自然にクーポン提案される
- 完了が視覚的にも触覚的にも分かる
- すべてがネイティブ画面内で完結する

### 2. 新規アプリの北極星体験を見つける

PoCの目的は、画面を並べることではなく、今後の本開発で守るべき体験原則を見つけることです。

そのため、このPoCでは次の問いに答えられる状態を目指します。

- どの画面構造なら注文が楽しいまま完了まで進めるか
- どの演出は体験価値を上げ、どの演出はノイズになるか
- どこまでを一画面で見せ、どこで段階分割するべきか
- 注文、クーポン、お気に入りをどう統合すると気持ちよいか

## プロダクト仮説

このPoCで作るべきアプリは、`クーポンを見るアプリ` でも `会員登録させるアプリ` でもなく、`カレーの注文を考える時間そのものを楽しめるアプリ` です。

ユーザーは店頭や移動中に次のように考えています。

- 今日は何を食べようか
- いつもの組み合わせにしようか
- トッピングを変えてみようか
- 今使えるクーポンはあるか

この思考の流れを分断せず、注文完了まで一筆書きで進めることがPoCの中心仮説です。

## 想定する体験

### 体験の骨子

1. 店舗を選ぶ
2. ベースのカレーを選ぶ
3. トッピングや辛さ、量を気持ちよく調整する
4. 気に入った組み合わせはその場で保存できる
5. 最終確認で、その注文に適したクーポンを自然に提案する
6. 注文完了時に、明確な達成感がある

### 既存アプリとの違い

- 会員登録を先に要求しない
- Webに遷移しない
- クーポンを別機能として切り離さない
- 完了状態を曖昧にしない
- 設計の中心を「業務都合」ではなく「注文の楽しさ」に置く

## UX原則

### 1. Thinking In Context

店舗でメニューを見ながら考える自然な順番に合わせてUIを設計する。
フォームを埋めさせるのではなく、選ぶたびに次の選択肢が心地よく出てくる構成にする。

### 2. Native First

WebViewやブラウザ的なレイアウトは最小限にし、ネイティブの操作感を優先する。
スクロール、選択、画面遷移、シート、検索、フィードバックはネイティブ挙動を基準にする。

### 3. Progressive Disclosure

最初から全部を見せない。
ベースメニュー、カスタマイズ、保存、クーポン提案、完了確認を段階的に見せることで、認知負荷を下げる。

### 4. Reward The Decision

ユーザーが選んだこと自体に楽しさを返す。
トッピング追加、辛さ変更、お気に入り保存、注文完了に対して、軽快な反応と小さな達成感を返す。

### 5. No Anxiety At Finish

完了直前と完了直後の不安をなくす。
最後に何が確定したのか、何が保存されたのか、次に何をすればよいのかを明確にする。

## UIの方向性

### 全体のトーン

- カレーの温度感が伝わる温かい色設計
- 情報量は多くても圧迫感を出さないレイヤー設計
- 写真、質感、余白、階層で食欲と期待感を作る
- 触って楽しいが、注文速度は落とさない

### 最新UI技術の取り込み方

このPoCでは、最新のAppleプラットフォームのデザイン更新を参考にしつつ、見た目だけの演出ではなく `階層性` `調和` `一貫性` を重視します。

採用候補:

- Liquid Glass系の素材表現を使ったレイヤー設計
- 標準コンポーネントを活かしたネイティブなナビゲーションと検索
- 物理感のあるスプリングアニメーション
- 状態変化に連動する滑らかなトランジション
- 触覚フィードバックと視覚フィードバックの組み合わせ
- 選択内容が浮かび上がるようなフォーカス表現

非採用方針:

- 効果が強すぎて可読性を落とす表現
- カスタムしすぎてOS標準の気持ちよさを失うUI
- Web的なカード羅列をそのままアプリ化しただけの構成

## PoCで実装したい主要機能

### 1. 店舗選択

- 現在地起点ではなくても選べる
- 店舗を選んだ感覚が明確に残る
- 選択後は注文体験が自然に始まる

### 2. メニュー選択とカスタマイズ

- ベースカレー選択
- トッピング追加
- 辛さ変更
- ライス量変更
- 合計価格のリアルタイム反映

### 3. お気に入り保存

- 注文途中または確定前に保存できる
- 名前を付けて保存できる
- 保存済みの組み合わせから即再開できる

### 4. クーポン提案

- 最終確認画面で自動提案する
- 適用可能な候補だけを見せる
- `使えるかどうか分からない` 状態を作らない

### 5. 注文完了演出

- 注文が確定したと一目で分かる
- 視覚、文言、アニメーション、触覚で完了を伝える
- 次の行動が明確である

## PoCの画面案

1. Welcome / Store Select
2. Menu Discovery
3. Curry Detail / Customize
4. Saved Combos
5. Order Review
6. Coupon Suggestion Sheet
7. Order Complete

## 画面ごとの狙い

### Store Select

- 注文の入り口を軽くする
- `どこで受け取るか` を決めたらすぐ食事の検討に移れるようにする

### Menu Discovery

- 一覧から探すだけでなく、気分で選べる見せ方を検証する
- 定番、おすすめ、前回系の見せ分けを試す

### Curry Detail / Customize

- このPoCの中心画面
- 選ぶ楽しさと分かりやすさの両立を検証する
- 選択結果が価格とビジュアルに即時反映されることが重要

### Order Review

- 注文の抜け漏れ不安を消す
- 合計、内容、受取店舗、お気に入り保存、クーポン提案への導線をここで整理する

### Order Complete

- 成功を曖昧にしない
- `注文できた` だけでなく `これで受け取れる` まで伝える

## 実装方針

### 技術前提

- iOSネイティブアプリのPoCとして設計する
- ネットワーク通信は必須にしない
- ダミーデータ、ローカル状態、モック注文完了で成立させる

補足:

- Liquid Glass などの最新UI表現を自然に活かしやすいよう、PoCは `iOS-first` で進める
- Android 展開はこのPoCの直接対象ではなく、体験原則の横展開対象として後続で判断する

### 実装の考え方

- バックエンド前提の画面ではなく、状態遷移前提の画面を作る
- データ構造は最小限でも、体験フローは実際に操作できる状態にする
- 画面単位のモックではなく、`一連の注文フロー` が通ることを重視する

### ローカルデータの扱い

PoCではバックエンドを持たないため、以下は端末内データまたはダミーデータで扱う。

- 店舗一覧
- メニュー一覧
- 商品詳細と価格
- クーポン候補
- お気に入り組み合わせ
- 注文完了状態

あわせて、PoCでは次の前提を置く。

- お気に入り保存はローカル保存で成立させる
- クーポン提案はローカルルールで算出する
- 注文完了はモック成功レスポンスとして扱う
- ログイン状態は前提にせず、ゲスト体験でも主要フローが完走できるようにする

## 成功条件

このPoCが成功と言える条件は次の通りです。

1. 注文フローを通したときに、既存アプリより明らかに気持ちよい
2. クーポン提案、お気に入り保存、完了演出が自然につながる
3. Webではなくネイティブである意味が明確に伝わる
4. 実サービス連携前でも、本開発したくなる方向性が見える

## 非機能上の目標

- 主要遷移が軽快である
- タップ反応が明確である
- アニメーションが長すぎない
- 可読性と操作性を犠牲にしない
- アクセシビリティを壊さない
