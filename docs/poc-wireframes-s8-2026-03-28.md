## 目的

この資料は、注文完了を `成功したと分かる` だけでなく、`このあと何をすればよいか分かる` 状態まで設計するためのワイヤーです。

PoCにおける完了画面の役割は次の3つです。

- 注文成功を曖昧にしない
- 受取情報をすぐ理解させる
- 次の行動へ迷わず進ませる

## 前提

- 実注文APIではなくモック成功レスポンス前提
- `S8 Order Complete` は full-screen push
- `Place Order` 押下後は短い処理演出を経て `S8` に遷移する
- 完了画面ではエラー分岐を扱わない

## S8 Order Complete

### 画面の役割

- 完了を視覚、文言、触覚で伝える
- 受取情報を明快に見せる
- メニュー再訪、Saved Combos 再訪、店舗変更の3導線を整理する

### 情報構造

1. 完了演出
2. 注文完了メッセージ
3. 受取情報
4. 今回の注文サマリー
5. 次の行動CTA

用途:

- 完了直後にも保存導線を残したい場合の派生案

扱い:

- 初版PoCでは必須にしない
- `S5` までに保存導線があるため、優先度は低い

## S8 コンポーネント意図

- `Success Mark`
  - 一目で完了が分かる中心要素
- `Pickup Info Card`
  - 受取先と受取時間を誤解なく伝える
- `Reference ID`
  - 実運用ではないが、注文が記録された感覚を作る
- `Your Order Card`
  - 最終的に何を注文したかを短く再確認させる
- `Next Actions`
  - 再利用や再探索の導線を整理する

## S8 重要ルール

- 最初に見せる情報は `成功` と `受取` の2つだけに絞る
- 受取店舗、受取時間、注文内容は必ず一画面で確認できる
- `Browse Menu Again` を主CTAにする
- `View Saved Combos` は副CTAだが十分目立たせる
- `Change Store` は破壊的ではないが、完了後の文脈では最も優先度を下げる

## 推奨CTA優先度

1. `Browse Menu Again`
2. `View Saved Combos`
3. `Change Store`

理由:

- 完了直後は次の注文か、保存済み構成の再利用が自然
- 店舗変更は完了後のメイン欲求ではない

## 状態差分

### S8 の主要状態

1. 成功演出中
2. 情報表示状態
3. Saved Combo 派生導線あり

### 推奨遷移

```text
S5 Place Order tapped
  -> brief loading / transition
  -> S8 success animation
  -> S8 settled information state
```

## アニメーションと反応の最低要件

- `Place Order` 押下後は 0.4秒前後の短い反応を返す
- S8 到達時は success mark が軽く拡大して止まる
- 触覚フィードバックは成功時に1回
- 成功演出は長くても1.2秒以内に情報表示状態へ移る

## アクセシビリティ注意点

- 完了状態はアイコンだけでなく文言でも示す
- 受取店舗、時間、注文内容、金額は VoiceOver で順に追える構造にする
- CTAは縦積みで押し間違いにくくする
- success animation 中でも Reduce Motion 環境では静的表示へ切り替えられるようにする

## 初版PoCで入れるもの / 後回しにするもの

### 初版PoCで入れる

- 成功演出
- 受取情報表示
- 注文サマリー表示
- `Browse Menu Again`
- `View Saved Combos`

### 後回しにしてよい

- 完了画面上での Save Combo 派生導線
- 共有導線
- カレンダー追加
- 通知設定導線
