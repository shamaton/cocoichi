# CoCo壱番屋アプリ 改修論点と新規アプリ計画

- 作成日: 2026-03-28
- 入力資料: `docs/appstore-review-analysis-611124786-jp-2026-03-23.md`
- 対象レビュー: 500件
- 目的: 既存アプリの改修優先度を整理し、新しいアプリを作る場合の計画判断に使う

## 要約

レビューから見える本質は、UIの好みではなく `注文成立の信頼性不足` と `アプリ体験の分断` です。

現状は次の3点が同時に起きています。

1. 注文が途中で失敗し、再試行も分かりにくい
2. アプリ会員とモバイルオーダー会員が分断されている
3. クーポンや会員登録が注文体験をむしろ悪化させている

このため、既存アプリの改善だけでは足りず、`注文を中心に据えた新アプリへの再設計` を前提に考えるべきです。
ただし、全面リプレイスの前に既存アプリ側で止血すべき論点も明確です。

## レビューから読み取れる構造問題

### 1. ユーザーの期待と実装がズレている

ユーザーは「アプリなら注文が早く、会員情報も引き継がれ、クーポンも自然に使える」と期待しています。
しかし実際には次の体験になっています。

- アプリから注文を始めてもWebに遷移する
- Web側で再ログインを求められる
- 注文や決済の途中でエラーになる
- クーポンが使える条件が分かりにくい
- 会員登録の完了前に店舗設定やメール認証で詰まる

その結果、ユーザーは「使いにくい」ではなく「アプリの存在理由がない」と判断しています。

### 2. 低評価の主因は機能不足ではなく失敗率の高さ

高評価レビューでは、クーポン、スクラッチ、モバイルオーダー自体の需要は確認できます。
つまり価値仮説は間違っていません。

問題は、価値を届ける前に以下で失敗していることです。

- 決済完了前に落ちる
- 注文内容が欠落する
- 受取時間が途中で無効になる
- 注文確認や履歴確認がアプリ内で完結しない

### 3. 業務都合の設計が前面に出ている

レビューでは、ユーザーが運用の都合を押し付けられているように感じる箇所が多く出ています。

- クーポン利用のためだけに店員を呼ぶ必要がある
- アプリ会員と注文会員が別である
- 店舗選択やお気に入り登録が先に要求される
- 注文方式の選択を複数回求められる

この状態では、機能追加よりも `体験の責務整理` が先です。

## 既存アプリの改修すべき点

以下は「新アプリを作るとしても先に直すべきもの」と「新アプリでは作り直すべきもの」を分けて整理しています。

### A. 既存アプリで即時に止血すべき項目

#### P0. 注文失敗と決済失敗の是正

- 注文作成から決済完了までの状態遷移を再設計する
- 二重送信、途中離脱、決済リトライ、失敗復旧を idempotent に扱う
- 注文が失敗したときに、`未確定` `決済失敗` `受付済み` を明示する
- `注文中` のままロックされて再注文できない不具合を解消する
- カート内容、受取条件、入力済み情報を端末側でも保持する
- アプリ内で注文番号と注文状態を確認できるようにする

#### P0. 認証分断の解消

- アプリ会員とモバイルオーダー会員を最低でもSSOで接続する
- アプリからWebビューへ遷移しても再ログインを要求しない
- 電話番号登録、メール登録、会員番号ログインの仕様を一本化する
- ユーザー自身が「どの会員状態なのか」をマイページで確認できるようにする

#### P0. 会員登録の簡略化

- 初回登録時の必須項目をメールアドレス、または電話番号中心に絞る
- お気に入り店舗登録を任意化する
- メール認証URL失敗時の再送、ブラウザ起動失敗時のフォールバックを入れる
- パスワード制約を現代的な標準へ改め、入力中にルールを即時表示する

#### P1. 店舗検索と受取時間の改善

- 郵便番号検索、現在地検索、店舗名検索を別入口で提供する
- GPS失敗時でも手入力検索が確実に使えるようにする
- 受取時間の在庫・受付制御は確認画面で最終確定する
- 選択不可になった場合は代替候補を自動提示する

#### P1. クーポン導線の再設計

- クーポンに `使える注文方式` `併用条件` `提示タイミング` を先に表示する
- モバイルオーダー対象クーポンは注文フロー内で自動表示する
- 店内タブレット、テイクアウト、事前決済で適用条件が違うならUIを分離する
- クーポン画面だけで完結する説明をやめ、利用場面ごとに文脈表示する

#### P1. パフォーマンスと観測性の強化

- 画面遷移時間、API失敗率、決済失敗率、離脱ポイントをイベント計測する
- クラッシュログ、WebView失敗、認証失敗、店舗検索失敗を監視する
- `注文開始 -> カート投入 -> 決済到達 -> 注文確定` のファネルを継続監視する

### B. 既存アプリでは延命せず、新アプリで再設計すべき項目

- ネイティブ内で注文完結する情報設計
- 会員、ポイント、クーポン、注文履歴の単一アカウント化
- 再注文、履歴、お気に入り導線を中心にしたリピート体験
- 店内注文とテイクアウト注文の責務分離
- 注文状況追跡、受取番号表示、通知連携のアプリ内完結

### C. 改修か新規かの判断マトリクス

| 論点 | 既存アプリで対応 | 新規アプリで対応 | 判断理由 |
| --- | --- | --- | --- |
| 決済失敗、注文失敗、再注文不能 | 必須 | 必須 | 売上毀損と離脱に直結するため、止血と再設計の両方が必要 |
| Web遷移後の再ログイン | 必須 | 原則廃止 | 既存でも改善余地はあるが、新規ではネイティブ完結が前提 |
| 会員種別の分断 | 部分改善 | 根本解決 | SSOで応急処置は可能だが、根本は単一アカウント化 |
| 会員登録の必須項目過多 | 必須 | 必須 | 既存でも削減可能で、新規でも登録設計の再定義が必要 |
| 店舗検索/GPS/郵便番号検索の詰まり | 必須 | 必須 | 既存の不具合修正対象であり、新規では探索導線から見直すべき |
| クーポンの分かりにくさ | 必須 | 必須 | 表示改善は既存で可能だが、適用体験は新規で作り直した方が良い |
| 注文履歴/受取番号確認のアプリ内完結 | 可能なら一部 | 必須 | 既存への継ぎ足しより、新規で状態管理ごと設計した方が整合する |
| 再注文中心のリピート体験 | 優先低 | 必須 | 止血後に効く成長要素であり、新規MVPの差別化要素になる |
| ポイント/特典常設化 | 優先低 | 段階対応 | 基本体験が安定する前に強化しても逆効果になりやすい |

## 新しいアプリで目指すべき体験

### プロダクト原則

1. 注文は3分以内に完了できる
2. ログインは一度で済む
3. クーポンは「見せる」ではなく「適用される」ものにする
4. 店舗都合の複雑さをアプリ画面に持ち込まない
5. エラー時でも何が起きたか分かり、復旧できる

### コアユーザージョブ

- 近くの店舗を選び、迷わず注文したい
- 前回と同じ注文をすぐ再注文したい
- 利用可能なクーポンや特典を自然に使いたい
- 注文できたかどうかを不安なく確認したい

## 新規アプリのMVP定義

### MVPに含めるもの

- 単一アカウントの会員基盤
- ネイティブ内で完結するテイクアウト注文
- 注文履歴と再注文
- 店舗検索
- クーポン自動適用または候補提示
- 注文完了後の受取番号、受取予定時刻、注文状態表示
- 決済失敗時の再試行導線
- 通知連携

### MVPから外してよいもの

- ゲーム的な施策の拡張
- 複雑な会員ランク制度
- 店内注文とテイクアウト注文の完全統合
- 多すぎる認証方式の併存

先に `注文成功率` と `再注文率` を作り、その後に販促を積む順序が妥当です。

## 新規アプリの要件整理

### 1. 機能要件

- 会員登録、ログイン、パスワードレスまたは標準的な認証
- 店舗検索、営業時間、受取可否判定
- 商品選択、トッピング、辛さ・量などのカスタマイズ
- カート保存
- 注文確認
- 事前決済
- 注文状態確認
- 注文履歴、再注文、お気に入り
- クーポン適用
- ポイント連携

### 2. 非機能要件

- 主要注文フローの画面応答を高速化する
- 注文APIは冪等であること
- 決済失敗や通信断から復帰可能であること
- 障害発生時にサポートが追跡できる注文IDを全件付与する
- アプリとサーバーのイベントログを紐付けて調査できること

### 3. 運用要件

- クーポンやキャンペーン条件を管理画面で更新できること
- 店舗ごとの受付可能時間や販売可否を柔軟に制御できること
- CSが注文失敗の問い合わせに即応できること

## KPI案

### 既存アプリの止血KPI

- 注文完了率
- 決済失敗率
- WebView遷移後の離脱率
- 会員登録完了率
- 注文関連問い合わせ率

### 新規アプリの立ち上がりKPI

- 初回注文完了時間
- 再注文率
- クーポン適用率
- 月次アクティブ注文者数
- アプリ経由注文比率

## 推奨する開発アプローチ

### フェーズ0. 企画と業務整理

- 現行の会員種別、注文種別、決済フロー、クーポン仕様を棚卸しする
- Web注文基盤を流用できる範囲と、刷新が必要な境界を明確にする
- 店舗運用に依存している制約を洗い出し、アプリ側で吸収すべきものを決める

成果物:

- 体験設計方針
- アカウント統合方針
- 注文状態遷移図
- クーポン適用ルール整理

### フェーズ1. 止血リリース

- 既存アプリ側で注文失敗、再ログイン、登録詰まりを優先修正する
- 同時に計測基盤を入れて、どこで失敗しているかを見える化する

成功指標:

- 注文失敗率の大幅改善
- 会員登録完了率の改善
- WebView遷移後の離脱率低下

### フェーズ2. 新規アプリMVP開発

- 単一アカウント基盤
- テイクアウト注文
- 注文状態表示
- 再注文
- クーポン適用

成功指標:

- 注文完了率
- 初回注文完了時間
- 再注文率
- サポート問い合わせ率

### フェーズ3. 段階移行

- 既存アプリから新アプリへ段階的に送客する
- 一部機能を新アプリ先行で提供し、利用実績を見ながら切り替える
- アカウント移行、ポイント移行、履歴移行をできるだけ自動化する

### フェーズ4. 成長施策

- パーソナライズされた再注文
- 会員特典の常設化
- ポイント・キャンペーン最適化

## 企画判断としての結論

### 判断1. 既存アプリは全面維持ではなく止血対象

既存アプリに対しては、まず `注文失敗` `認証分断` `登録詰まり` を修正しない限り、広告や販促を強めても逆効果です。

### 判断2. 新規アプリは「注文アプリ」として再定義する

現状はクーポンアプリ、会員アプリ、注文アプリが中途半端に混ざっています。
新規アプリは `注文成功` を中心価値に据え、会員・クーポン・ポイントはその周辺に整理すべきです。

### 判断3. 新規開発の前提条件はバックエンド統合

新アプリだけ作り直しても、会員基盤と注文基盤が分断したままでは同じ問題を再生産します。
最優先の設計論点はUIではなく `アカウント統合` と `注文状態管理` です。

## 次に詰めるべき論点

1. アプリ会員とモバイルオーダー会員をどう統合するか
2. 既存Web注文基盤を流用するか、注文APIを新設するか
3. 決済失敗時の正規状態遷移をどう定義するか
4. クーポン適用ルールをどの画面責務に置くか
5. 新規アプリMVPで店内注文まで含めるか、テイクアウトに絞るか

## 意思決定用の提案

現時点では、次の方針が妥当です。

1. まず既存アプリのP0止血を1スプリントで実施する
2. 並行して新規アプリの要件定義を開始する
3. 新規アプリの初版は `テイクアウト注文の成功体験` に集中する
4. 会員、注文、クーポン、ポイントの責務整理が終わるまで、周辺施策の拡張は後回しにする
